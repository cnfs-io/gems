## What This Is

This is the `pcs` CLI gem — an infrastructure bootstrapping and management tool for bare-metal private cloud sites. It is a Ruby gem using `dry-cli` and `tty-prompt` that runs on a Raspberry Pi to bootstrap sites and manage Proxmox clusters, TrueNAS storage, inter-site networking, and tunnel exposure.

## What This Is NOT

- This is NOT the PCS Rails application (tenant management UI). That is a separate project.
- This is NOT the colo gem (`~/.local/share/ppm/gems/colo`). The colo gem handles L1 concerns (UniFi networking, physical site networking via Terraform). PCS is exclusively L2 (cloud provider) focused.
- Do not create Rails files, engines, or web-related code. This gem is a pure CLI tool.

## Project Structure

PCS follows a project-scoped pattern similar to Rails. All configuration, site data, and state live within a single project directory. A project is created with `pcs new` and detected by the presence of `config/pcs.yml`.

```
rws-pcs/                          # project root, created by `pcs new`
  .env                            # gitignored — PCS_SITE=roc (current site selection)
  .env.example                    # checked in — documents the convention
  .gitignore
  Gemfile                         # pins gem versions, declares provider gems
  README.md

  config/
    pcs.yml                       # main project config (providers, defaults, conventions)

  sites/
    roc/
      ipam.yml                    # network intent (generated by pcs bootstrap)
      cluster.yml                 # cluster config (generated by pcs cluster create)
    sg/
      ipam.yml
      cluster.yml

  states/
    roc/
      state.yml                   # device + service runtime state
    sg/
      state.yml
```

### Project Detection

PCS detects the project root by walking up from CWD looking for `config/pcs.yml`. This is the same pattern Rails uses with `config/application.rb`.

### Site Selection

The current site is determined by (in priority order):
1. `PCS_SITE` environment variable (explicit, e.g., from mise, shell, CI)
2. `.env` file at project root (loaded via dotenv gem)
3. Interactive prompt if neither is set

The `.env` file is gitignored. A `.env.example` is checked in to document the convention.

## Scope — The Two Verticals

PCS manages two concerns, each with a pluggable provider:

| Vertical | Default Provider | Purpose |
|----------|-----------------|---------|
| `cluster` | Proxmox VE | Hypervisor cluster formation and management |
| `nas` | TrueNAS SCALE | Network-attached storage setup and management |

Providers are configured in `config/pcs.yml`. The CLI commands are generic (e.g., `pcs cluster create` not `pcs proxmox create`), but the underlying implementation delegates to the configured provider.

## Architecture

```
CLI layer (dry-cli commands) → Services (business logic) → Providers (vendor adapters)
                                    ↓
                              Models (Config, IPAM, State)
                                    ↓
                              Host Strategies (PVE, TrueNAS, PiKVM, RPi)
```

### Provider Architecture

Provider-specific code lives under `lib/pcs/providers/<n>/`. Each provider implements the interface expected by its vertical. For now, there is one provider per vertical. The adapter interface will be extracted when a second provider materializes — don't over-abstract.

```
lib/pcs/providers/
  proxmox/          # cluster provider
  truenas/          # nas provider
```

## Command Tree

### Project Management

```
pcs new <name>                    # Scaffold a new PCS project
pcs use <site>                    # Set the current site (writes .env)
```

### Bootstrap Phase (site initialization + device lifecycle)

```
pcs bootstrap                     # Bootstrap RPi as L2 control plane for current site
pcs bootstrap discover            # Scan network, SSH-detect OS types, find PiKVMs
pcs bootstrap provision <host>    # Configure static IPs, hostname, SSH keys on a device
pcs bootstrap watch               # PXE boot event loop for new hosts
pcs bootstrap status              # Show bootstrap progress (devices and readiness)
pcs bootstrap next                # Guided "what should I do now?" workflow
```

### Service Phase (runtime operations)

```
pcs cluster create                # Form cluster from provisioned hosts
pcs cluster status                # Cluster health and membership

pcs nas create                    # Set up NAS pools/datasets
pcs nas status                    # NAS health

pcs status                        # Unified view: all devices + all services
pcs version                       # Print version
```

## Current Implementation Focus

**Phase 0: Project scaffolding** — `pcs new`, `pcs use`, project detection, site resolution

**Phase 1: Bootstrap** ← PRIMARY FOCUS

Implement the `pcs bootstrap` command family. This is the device lifecycle: taking bare-metal hosts that already have L1 network connectivity (DHCP from the colo/UniFi layer) and preparing them for service configuration.

The bootstrap phase assumes:
- L1 (colo gem) has already configured the physical network (VLANs, DHCP, etc.)
- Devices have DHCP-assigned IPs on the servers network
- The RPi is connected and has the `pcs` gem installed
- A PCS project exists and the operator has `cd`'d into it

**Phases 2-5** (network, cluster, nas, tunnel) will be tackled sequentially after bootstrap is solid and tested.

## Key Design Decisions

- **Project-scoped** — all config, site data, and state live within the project directory. No `~/.local` or `~/.config` paths.
- **L1 (colo gem) owns DHCP.** dnsmasq runs as proxy PXE only — never as a DHCP server.
- **Static IPs are configured directly** on hosts via SSH or API. No DHCP reservations.
- **`pcs bootstrap` runs from the project root.** Project detected by `config/pcs.yml`.
- **Site selected via `.env`** — `PCS_SITE=roc` loaded by dotenv, overridable via env var.
- **Podman** is the container runtime, not Docker.
- **Generic command names** — `cluster`, `nas`, `network`, `tunnel` — not vendor-specific names.
- **Providers configured in `config/pcs.yml`** — provider-specific config in `config/<provider>.yml`.
- **Direct SSH** (`net-ssh` gem) for host management — not Ansible.
- **Faraday** for REST API calls (TrueNAS, Proxmox).

## Implementation Order

1. **Project scaffolding**: `pcs new`, `pcs use`, project root detection, site resolution via dotenv
2. **Models**: Config (reads from `config/pcs.yml`), IPAM (reads from `sites/<site>/ipam.yml`), State (reads from `states/<site>/state.yml`)
3. **Adapters**: SSH, Nmap, Dnsmasq, SystemCmd
4. **Host strategies**: Base, RPi, PiKVM, TrueNAS, PVE
5. **Services**: BootstrapService, DiscoverService, ProvisionService, WatchService
6. **CLI commands**: Wire services to dry-cli commands under `bootstrap` namespace
7. **Cross-cutting**: StatusService (unified view), NextService (guided workflow)

## Code Conventions

- **FlatRecord models use method access, not hash access.** All model objects (Host, Site, Config, Service, etc.) expose attributes via methods (`host.compute_ip`), never hash-style (`host[:compute_ip]`). This applies everywhere: Ruby code, ERB templates, specs. FlatRecord does not implement `[]` — hash access will raise NoMethodError. If you see `[:attribute]` on a model object anywhere in the codebase, it is a bug from the pre-FlatRecord era and should be changed to `.attribute`.
- Use `Pathname` instead of `File`, `FileUtils`, or `Dir` for path operations
- Use `ENV.fetch('VAR')` instead of `ENV['VAR']`
- Use `frozen_string_literal: true` in all files
- Module namespace is `Pcs` (not `PCS`) to match existing code
- Ruby 3.1+ features are fine (pattern matching, endless methods, etc.)
- Keep methods short and focused
- No minitest/rspec for the spike — but code should be well-factored and testable

## What Was Removed

The `lib/pcs/network/` directory (UniFi/Terraform commands: init, plan, apply, show, destroy) has been extracted to the **colo gem** at `~/.local/share/ppm/gems/colo`. That code manages L1 concerns and is no longer part of PCS.

If you see references to `Pcs::Network::Init`, `Pcs::Network::Plan`, etc. in old code or docs, those belong to the colo gem now.
